用户看到的积分余额是一个数字，例如 10,000 积分。但如果系统内部也只把它作为一个简单的 balance 来管理，就很难准确执行退款、过期和商品限制等实际策略。

ZDP Money Platform 以积分批次（credit lot）为单位管理积分。用户看到的数字是多个批次的总和，但实际扣减时，系统仍会检查每个批次的来源、过期时间、是否可退款以及适用的商品范围。

本文结合生产环境中的 Rust 代码，说明系统如何决定各批次的扣减顺序，以及为什么不能静默跳过过期批次。

## 为什么要把余额拆分成积分批次

如果只用一个数字管理积分，扣减本身很简单。从 balance 中减去 amount，看起来就完成了。真正的问题会在之后出现。

收到退款请求时，系统很难可靠地判断其中有多少是用户实际购买的金额。促销积分过期时，也很难确定应该删除哪一部分价值。此外，系统无法区分仅限特定商品使用的促销积分和普通购买积分。

因此，ZDP 会把同样的 10,000 积分保存为多个来源和属性不同的批次。

## 积分批次示例

| 批次 ID | 用途 | 剩余积分 | 过期时间 | 可退款 |
| --- | --- | --- | --- | --- |
| signup_free | 注册奖励 | 3,000 | 2026-07-31 | 否 |
| paid_bonus | 购买赠送 | 2,000 | 2026-08-31 | 否 |
| paid_base | 实际购买 | 5,000 | - | 是 |

在这种模型中，扣减顺序本身就是产品策略。通常，先使用即将消失的价值，例如即将过期或不可退款的积分，并尽可能延后扣减可退款本金，对用户更有利。

当前实现按照以下顺序排列候选批次。

策略优先级 → 过期时间较早者 → 创建时间较早者 → 批次 ID

## 为什么跳过过期批次很危险

很多人一开始都会提出同一个问题。

“为什么不把过期批次从候选项中移除，然后使用下一个批次？”

这种模式很危险，因为它会把数据异常造成的成本悄悄转嫁给用户。

如果一个免费批次已经超过过期时间，却仍然处于 Active 状态，说明过期工作流发生了延迟，或者账本没有得到正确修正。静默跳过这个批次，会使系统转而扣减后面的已购买积分。

支付虽然成功了，但用户损失的是自己直接购买的价值，而不是使用本应已经完成状态处理的免费积分。

因此，当前实现会明确返回失败。

```rust
if lot.lot_status == CreditLotStatus::Active
    && lot.remaining_amount_credit_unit > 0
    && lot_allows_product_group(lot, &request.product_group)
    && lot.expires_at.as_deref().is_some_and(|expires_at| {
        credit_lot_expires_at_or_before(expires_at, &request.occurred_at)
    })
{
    return Err(CreditLotSpendError::ExpiredLot {
        credit_lot_id: lot.credit_lot_id.clone(),
    });
}
```

ExpiredLot 不是普通的用户错误。它是一道保护措施，用于保留“过期处理与实际账本状态已经不一致”这一信号。上层可以根据该错误触发过期修正，或选择明确的重试策略。

## 为什么规划阶段和存储阶段要分别校验

积分扣减主要分为两个阶段。

规划阶段决定请求的金额应从哪些批次中扣减。存储阶段把这个决定实际写入数据库。

计划有效，并不保证开始持久化时数据仍然完全相同。另一个请求可能已经先消费了同一个批次，或者计划中的过期快照可能已经陈旧。

因此，持久化边界会在存储前再次校验过期快照。

```rust
if consumption
    .expires_at_at_consumption
    .as_deref()
    .is_some_and(|expires_at| expires_at <= input.plan.occurred_at.as_str())
{
    return Err(CreditLotSpendStorageError::PlanMismatch(
        "expires_at_at_consumption",
    ));
}
```

这两项检查的目的不同。第一项是领域规则，用于判断“这个批次能否成为扣减候选项？”。第二项是存储边界规则，用于确认“即将写入的计划是否与自身数据相矛盾？”。

前者防止错误决策，后者防止错误记录。

## 从边界时间点开始即视为过期

如果某个批次在 2026-07-31 00:00:00 过期，那么恰好发生在这一时刻的扣减必须被拒绝。当前实现只要发现 expires_at 小于或等于 occurred_at，就会把该批次视为已过期。

测试明确固定了这一边界。

```rust
assert_eq!(
    plan_credit_lot_spend(sample_credit_spend_request(1), &[expires_at_usage]),
    Err(CreditLotSpendError::ExpiredLot {
        credit_lot_id: "lot_expires_at_usage".to_owned()
    })
);
```

当前比较依赖字符串的字典序。只有当所有时间戳都采用相同的规范 UTC 表示时，这种做法才安全。不同的时区偏移或未规范化的小数秒都可能导致字符串比较出错。

## 这项修改不能解决什么

过期校验并不能解决所有问题。

并发问题：两个请求可能读取同一份快照，并在同一时间尝试消费同一个批次。数据库必须通过原子条件更新、乐观锁或合适的锁定策略阻止这种情况。

运行恢复：ExpiredLot 错误增加时，不应把它们当作普通用户错误而忽略。系统必须能够判断根因是过期任务延迟、事件丢失，还是无效的状态转换。

明确返回失败的最终目的，是让问题能够被观察和修复。

## 结语

积分扣减看起来像一个简单的减法函数，但实际上是一项策略：决定哪一种价值先被使用，哪一种价值应当保护到最后。

仅仅一行静默跳过过期免费批次的代码，就可能消耗用户购买的余额。因此，我们选择在状态无效时准确停止，而不是强行让支付成功。

我们相信，这个选择终有一天会在真实的生产事故中保护我们。
