用户看到的是一个余额：10,000 积分。但在支付系统内部，如果也把这个数字当作一个不加区分的余额，很快就会出现问题。3,000 积分的注册奖励可能在本月过期，购买的 5,000 积分可能可以退款，另外 2,000 促销积分可能只能用于指定商品。

因此，ZDP Money Platform 会以积分批次（credit lot）的形式保存余额。用户看到的数字是所有批次的总和，但在扣减时，仍需考虑每个批次的来源、过期时间、是否可退款以及适用的商品范围。

## 为什么一个余额需要多个批次

只保存一行余额，扣减确实很简单：减去金额，然后继续。麻烦会在之后出现。收到退款请求时，系统无法可靠地区分购买资金和促销赠送；执行过期任务时，也无法判断余额中的哪一部分应该消失。

采用批次模型后，同样的 10,000 积分可以表示为：

| 批次 | 用途 | 剩余积分 | 过期时间 | 可退款 |
| --- | --- | --- | --- | --- |
| signup_free | 注册奖励 | 3,000 | 7月31日 | 否 |
| paid_bonus | 购买赠送 | 2,000 | 8月31日 | 否 |
| paid_base | 实际购买 | 5,000 | 永不过期 | 是 |

扣减顺序由此成为产品策略。通常，对用户更有利的做法是先使用即将消失的价值，把可退款的本金保留到最后。我们的实现会按照优先级、过期时间、创建时间和批次 ID 对候选项排序。

## 为什么静默跳过过期批次很危险

最直观的实现是从候选列表中移除过期批次，然后继续处理下一个批次。

但这种做法可能让用户承担代价，从而掩盖数据问题。如果一个免费批次已经过期，却仍处于 Active 状态并保留正余额，说明过期工作流或账本修正发生了延迟。静默跳过它，会让系统转而扣减后面的已购买积分。支付虽然成功了，但用户花掉了真实购买的价值，而状态异常的促销积分仍未得到解释。

当前实现会在遇到处于 Active 状态、余额为正、适用于当前商品且已经过期的批次时明确失败。

```rust
if lot.lot_status == CreditLotStatus::Active
    && lot.remaining_amount_credit_unit > 0
    && lot_allows_product_group(lot, &request.product_group)
    && lot.expires_at.as_deref().is_some_and(|expires_at| {
        credit_lot_expires_at_or_before(expires_at, &request.occurred_at)
    })
{
    return Err(CreditLotSpendError::ExpiredLot {
        credit_lot_id: lot.credit_lot_id.clone(),
    });
}
```

这种失败并不是为了防御而增加的不便。它不会自动转向付费价值，而是暴露过期处理与账本状态之间的不一致。上层调用方随后可以修正过期状态，或有意识地重试。

## 为什么还要在存储边界再次校验

扣减领域逻辑会先生成一个计划，描述应该从哪些批次扣减。计划有效，并不意味着开始持久化时其中的数据仍然有效。另一个请求可能已经消费了该批次，或者计划携带的过期快照本身不一致。

因此，持久化规划器会再次检查过期快照。

```rust
if consumption
    .expires_at_at_consumption
    .as_deref()
    .is_some_and(|expires_at| expires_at <= input.plan.occurred_at.as_str())
{
    return Err(CreditLotSpendStorageError::PlanMismatch(
        "expires_at_at_consumption",
    ));
}
```

这两项检查承担不同职责。第一项是领域规则，用于决定哪些批次可以成为扣减候选项。第二项是存储边界的不变量，用于拒绝与自身快照相矛盾的计划。前者防止错误决策，后者防止错误记录。

## 从边界时间点起即视为过期

如果某个批次在 00:00:00 过期，那么恰好发生在 00:00:00 的扣减必须被拒绝。实现中，只要 `expires_at` 小于或等于 `occurred_at`，就会视为已过期；测试也固定了这一边界。

```rust
assert_eq!(
    plan_credit_lot_spend(sample_credit_spend_request(1), &[expires_at_usage]),
    Err(CreditLotSpendError::ExpiredLot {
        credit_lot_id: "lot_expires_at_usage".to_owned()
    })
);
```

当前比较依赖字符串的字典序。只有当所有时间戳都采用相同的规范 UTC 表示时，这种做法才是安全的。如果允许任意时区偏移，或允许采用不同规范化方式的小数秒，仅比较字符串是不够的。输入契约必须保持严格，或者在边界处先把值解析为真正的时间类型再进行比较。

## 这项修改不能解决什么

过期校验无法解决并发扣减。两个请求可能读取同一份快照，并尝试消费同一个批次。数据库仍然需要原子条件更新或锁定策略。规划与持久化之间的校验可以减少矛盾，但无法单独消除竞态条件。

运行恢复同样重要。ExpiredLot 错误数量上升时，不应把它们当作普通用户错误而掩盖。它应当指向延迟的过期任务、丢失的事件或无效的状态转换。明确失败的目的，就是让这种不一致能够被观察和修复。

积分扣减看起来像减法，但实际上是一项策略：决定哪一种价值先消失，哪一种价值应当受到保护。仅仅一行跳过状态异常的免费积分的代码，就可能烧掉用户购买的余额。在这种情况下，准确停止比强行让支付成功更好。
