사용자 화면에는 크레딧 잔액이 하나의 숫자(예: 10,000원)로 보입니다. 하지만 내부에서 이를 단순한 하나의 balance로만 관리하면, 환불·만료·상품 제한 등 실제 정책을 정확히 적용하기가 매우 어렵습니다.

ZDP Money Platform에서는 크레딧을 credit lot 단위로 관리합니다. 사용자가 보는 숫자는 여러 lot의 합계이지만, 실제 차감 시에는 각 lot의 출처, 만료 시각, 환불 가능 여부, 사용 가능한 상품 범위를 모두 확인합니다.

이번 글에서는 어떤 순서로 lot을 차감할지 결정하는 방법과, 만료된 lot을 왜 조용히 건너뛰면 안 되는지를 실제 Rust 코드와 함께 정리했습니다.

## 왜 잔액을 credit lot으로 쪼개서 관리해야 하는가

크레딧을 하나의 숫자로만 관리하면 차감 자체는 간단합니다. balance에서 amount를 빼면 끝입니다. 문제는 그 이후입니다.

환불 요청이 들어왔을 때 얼마가 실제로 충전한 금액이었는지 알기 어렵습니다. 프로모션 만료 시점에 어떤 금액을 제거해야 하는지 추적하기도 어렵습니다. 또한 특정 상품에만 사용할 수 있는 프로모션 크레딧과 일반 충전금을 구분할 수 없습니다.

그래서 ZDP에서는 같은 10,000원이라도 출처와 속성이 다른 여러 조각(lot)으로 보존합니다.

## credit lot 예시

| Lot ID | 성격 | 잔액 | 만료 시각 | 환불 가능 |
| --- | --- | --- | --- | --- |
| signup_free | 가입 보상 | 3,000 | 2026-07-31 | 불가 |
| paid_bonus | 충전 보너스 | 2,000 | 2026-08-31 | 불가 |
| paid_base | 실제 충전금 | 5,000 | - | 가능 |

이 구조에서는 차감 순서 자체가 제품 정책이 됩니다. 일반적으로는 먼저 사라질 금액, 즉 만료 예정이거나 환불할 수 없는 금액을 먼저 사용하고, 환불 가능한 원금은 최대한 늦게 차감하는 것이 사용자에게 유리합니다.

현재 구현에서는 후보 lot을 다음 순서로 정렬합니다.

우선순위(정책) → 만료 시각(빠른 순) → 생성 시각(오래된 순) → lot ID

## 만료된 lot을 건너뛰면 왜 위험한가

많은 사람이 처음에는 이렇게 생각합니다.

“만료된 lot은 후보에서 제외하고 다음 lot을 사용하면 되지 않을까요?”

이 방식은 데이터 이상을 사용자 비용으로 조용히 전가하는 위험한 패턴입니다.

상태가 Active인데 이미 만료 시각이 지난 무료 lot이 남아 있다면, 이는 만료 처리 작업이 늦었거나 원장 보정이 제대로 이루어지지 않았다는 신호입니다. 이런 lot을 조용히 건너뛰면 시스템은 뒤에 있는 유료 충전금을 대신 차감하게 됩니다.

결제는 성공하지만, 사용자는 원래 사라졌어야 할 무료 크레딧 대신 직접 충전한 금액을 잃게 됩니다.

그래서 현재 구현은 다음과 같이 명시적으로 실패를 반환합니다.

```rust
if lot.lot_status == CreditLotStatus::Active
    && lot.remaining_amount_credit_unit > 0
    && lot_allows_product_group(lot, &request.product_group)
    && lot.expires_at.as_deref().is_some_and(|expires_at| {
        credit_lot_expires_at_or_before(expires_at, &request.occurred_at)
    })
{
    return Err(CreditLotSpendError::ExpiredLot {
        credit_lot_id: lot.credit_lot_id.clone(),
    });
}
```

이 ExpiredLot 오류는 단순한 사용자 오류가 아닙니다. “만료 처리와 실제 상태가 어긋났습니다”라는 신호를 숨기지 않기 위한 방어 장치입니다. 상위 계층에서는 이 오류를 받아 만료 보정 작업을 트리거하거나 재시도 전략을 선택할 수 있습니다.

## 계획 단계와 저장 단계에서 각각 검사하는 이유

차감 로직은 크게 두 단계로 나뉩니다.

계획 단계에서는 어떤 lot에서 얼마를 사용할지 결정합니다. 저장 단계에서는 결정한 내용을 실제로 데이터베이스에 기록합니다.

계획을 세웠다고 해서 저장 시점의 데이터가 동일하다는 보장은 없습니다. 다른 요청이 먼저 같은 lot을 소비했을 수도 있고, 계획에 포함된 만료 스냅샷이 이미 오래된 상태일 수도 있습니다.

따라서 저장 직전에도 만료 스냅샷을 한 번 더 검사합니다.

```rust
if consumption
    .expires_at_at_consumption
    .as_deref()
    .is_some_and(|expires_at| expires_at <= input.plan.occurred_at.as_str())
{
    return Err(CreditLotSpendStorageError::PlanMismatch(
        "expires_at_at_consumption",
    ));
}
```

이 두 검사는 목적이 다릅니다. 첫 번째 검사는 도메인 규칙으로 “이 lot을 소비 후보로 삼아도 되는가?”를 판단합니다. 두 번째 검사는 저장 경계 규칙으로 “지금 기록하려는 계획이 스스로 모순되지 않는가?”를 확인합니다.

하나는 잘못된 결정을 막고, 다른 하나는 잘못된 기록을 막습니다.

## 경계값은 만료 시각부터 제외합니다

만료 시간이 2026-07-31 00:00:00이라면 정확히 그 시각에 발생한 사용은 허용해서는 안 됩니다. 구현에서는 expires_at이 occurred_at보다 작거나 같은 경우를 만료로 판단합니다.

이 경계 조건은 테스트로도 명확히 고정되어 있습니다.

```rust
assert_eq!(
    plan_credit_lot_spend(sample_credit_spend_request(1), &[expires_at_usage]),
    Err(CreditLotSpendError::ExpiredLot {
        credit_lot_id: "lot_expires_at_usage".to_owned()
    })
);
```

현재 비교는 문자열 순서에 의존하고 있습니다. 이 방식이 안전하려면 모든 시각이 동일한 canonical UTC 형식으로 들어온다는 계약이 반드시 지켜져야 합니다. 오프셋이 다르거나 소수점 표기가 정규화되지 않으면 문자열 비교는 틀릴 수 있습니다.

## 이 수정으로 끝나지 않는 부분

만료 검사를 추가했다고 해서 모든 문제가 해결되는 것은 아닙니다.

동시성 문제: 두 요청이 같은 스냅샷을 읽고 동시에 같은 lot을 차감하려는 상황은 데이터베이스의 원자적 조건부 갱신(optimistic locking)이나 분산 락으로 막아야 합니다.

운영 복구: ExpiredLot 오류가 증가한다면 이를 단순한 사용자 오류로 치부해서는 안 됩니다. 만료 작업 지연, 이벤트 유실, 잘못된 상태 전이 중 무엇이 원인인지 추적할 수 있어야 합니다.

실패를 명시적으로 만든 이유는 결국 문제를 관측하고 수정하기 위해서입니다.

## 마치며

크레딧 차감은 단순히 숫자를 빼는 함수처럼 보이지만, 실제로는 “어떤 금액을 먼저 사용하고, 어떤 금액을 끝까지 보호할 것인가”를 결정하는 정책입니다.

만료된 무료 lot을 조용히 건너뛰는 코드 한 줄이 사용자의 유료 잔액을 소진시킬 수 있습니다. 그래서 결제를 억지로 성공시키기보다, 상태가 올바르지 않을 때 정확히 멈추는 방식을 선택했습니다.

이 선택이 언젠가 운영 중에 우리를 구해 줄 것이라고 믿습니다.
